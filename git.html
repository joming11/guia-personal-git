Flujo Basico de Git

Estados de Git

  1. Working directory(modified)
  2. staging area(staged)
  3. commit (commited)
  4. Remote

el Working directory es el area de trabajo, la carpeta en si, donde inciamos el repositorio es donde se modificaran, eliminaran o agregaran archivos, por eso es el estado que esta en constante modificacion.

el Staging area, es el area donde estan los archivos que van a ser confirmados y enviados a HEAD la cabecera del repositorio, en este estado aun pueden ser devueltos al working directory para una nueva modificacion o para ser confirmados y guardarlos en el repositorio.

HEAD, es el estado donde estan confirmados los archivos

El estado remoto es cuando ya las carpetas se encuentran en un sistema como github, gitlab, bitbucket

Comandos para trabajar en el flujo basico de Git

git add archivo -> para agregar un elemento al staging

git add . -> para agregar mas de un elemento al staging

git commit para confirmar un archivo y enviarlo a la cabecera del repositorio al no tener la bandera -m se abrira el editor para definir un mensaje del commit

git commit -m para describir en un mensaje de la confirmacion

Github

para enlazar el repositorio local con un repositorio en github

se crea un nuevo repositorio en github, nombre y descripcion se agrega un readme y un .gitignore si no se han creado previamente y se agrega una licencia

Al crear un repositorio en github dara las indicaciones para crear el repositorio local desde la terminal. crear la carpeta, el readme, git init, git add, git commit, git branch -M main git remote add origin https://url git push -u origin main actualmente, la convencion es usar main como nombre de rama principal, por eso se hace el push -u origin desde main

si ya se ha creado el repositorio local, github dara las opciones para enlazarlo git remote add origin https://url para enlazar los repositorios git branch -M main git push -u origin main para subir los archivos

una vez que los repositorios, local y remoto esten enlazados y se hagan nuevos cambios para subirlos a github se utlizaria solo git push ya no seria necesario colocar -u origin

Si se realizara un cambio desde github, para bajarlo al repositorio local se usa git pull

Ramas en git (branch)

todo el desarrollo de un repositorio ocurre en una rama, por lo general es o era la rama master, por tema de inclusion, ahora la rama principal por convencion se llama main para pasar el desarrollo de la rama master a la rama main, hacemos: git branch -m master main -se crea la rama local y se le pasa el historial de la rama master git branch -m master main

Se hace un push de la nueva rama principal al repositorio remoto

se cambia el HEAD actual a la rama main git symbolic-ref refs/remotes/origin/HEAD refs/remotes/orign/main

para eliminar la rama master (si se desea) del repositorio remoto git push origin --delete master

Todo el cambio de rama se puede evitar colocando desde un principio git branch -M main

Ramas en git (branch)

para crear una rama -> git branch nombre-rama el nombre de la rama debe ser en minusculas, puede tener numeros y guion medio. no puede ser en mayusculas ni tener simboloso ni espacios en blanco.

git checkout nombre-rama -> para cambiar de rama

git checkout -b rama -> para crear una rama y cambiarse a ella

git checkout -d rama -> para eliminar una rama

git branch -D nombre-rama -> borrar forzosamente una rama

git branch -> para listar las ramas

git branch --no-merged -> ramas no fusionadas a la rama actual

git branch --merged -> ramas fusionadas a la rama actual

Rebasar ramas

git checkout rama-secundaria

git rebase rama-principal

Al crear una nueva rama y trabajar en ella, no se podran subir directamente al repositorio remoto, se tendra que usar otro comando push git push -u (--set-upstream) origin nombre-rama

git push origin --delete nombre-rama para eliminar una rama del repositorio remoto

Fusionar ramas

para fusionar dos ramas, nos situamos en la que queremos los cambios (el contenido fusionado)

ej: git checkout main git merge rama-con-cambios

existen dos escenarios principales

fast-forward: la fusion se hace automatica, sin conflictos por resolver

manual merge: la fusion se debe hacer manualmente para resolver conflictos de duplicacion de contenido ej: el mismo archivo tiene cambios distintos en ramas distintas al fusionar, git no sabe cuales cambios dejar se debe modificar en el archivo en conflicto directamente en el editor para definir cuales cambios se van a subir. github y vsc brinda botones interactivos sobre el codigo para ayudar en estos casos idealmente es preferible hacer estos arreglos antes de hacer un push

luego se sube el archivo, git add, git commit, git push

Cambios en los commits

git log -> permite ver el historial de los commits con sus ids

git log --online -> permite ver el historial de los commits en una linea

(HEAD-> main) cabecera del repositorio local

(origin/main, origin/HEAD) cabecera del repositorio remoto

se sale de esa lista usando la tecla q

git checkout + id de un commit nos lleva a un punto especifico del proyecto

si se modifica un archivo, se hace git add . y se hace un commit y nos damos cuenta que hay un error, o se debe agregar algo mas, se corrige el archivo, se hace git add . y no se usa el comando normal del commit, sino -> git commit --amend --no-edit para no crear un nuevo commit, sino, dejar el commit anterior

renombrar un commit -> git commit --amend -m "nuevo msj del commit"

git reset -hard HEAD-1 -> para eliminar el ultimo commit

Registro de hostorial

inicialmente se puede usar git log -> muestra el historial de commits con su id, fecha y desde que rama fue hecho

se puede usar git log --online muestra el historial de commits en una fila

se puede usar git log > commits.txt -> crea un archivo que contiene la lista de commits

se puede usar git log -n -> se sustituye la n por un numero entero y mostrara los n cambios recientes

se puede usar git log --after"año-mes-dia 00:00:00" ->muestra los cambios realizados despues de una fecha determinada

se puede usar git log --before"año-mes-dia 00:00:00" ->muestra los cambios realizados antes de una fecha determinada

git reflog -> muestra el registro de acciones del log, inserciones, cambios, eliminaciones, fusiones

git diff -> para ver diferencias entre el working directory y el staging area

git log --online --graph --all -> para ver graficamente el historial

Reseteo del historial

git status ayuda a observar en que estado esta uno o mas archivos

git reset permite deshacer un add y tambien un cambio en el HEAD

git reset --soft -> elimina el ultimo cambio en HEAD

git reset --mixed -> elimina HEAD y staging

git reset --hard -> elimina HEAD, staging y working directory

git reset id-commit -> elimina todos los cambios despues del commit indicado, preservando los cambios localmente

git reset --hard id-commit -> desecha todo el historial y regresa al commit indicado

Resetear un repositorio completamente

lo primero es posicionarse en la carpeta con el repositorio

mv .git/config ~/saved_git_config -> este comando indica tomar el archivo config del archivo oculto .git y guardarlo en la carpeta de usuario inicial con el nombre saved_git_config

rm -rf .git -> para borrar el .git

git init para inicializar un nuevo repo

git branch -M main -> para definir main como la rama principal

git add .

git commit -m "primer commit"

mv ~/saved_git_config .git/config -> comando para traer la configuracion remota

git push --forced origin main

Remotos

git remote -v -> muestra los origenes remotos con detalle

git remote add nombre-origen + https://url -> agregar un origen remoto

git remote rename nombre-anterior nombre-nuevo -> renombrar un origen remoto

git remote remove nombre-origen -> eliminar un origen remoto

git checkout --track -b rama-remota origin/rama-remota -> descargar una rama remota a local diferente a la principal

Etiquetas (Tags) Semver - versionado semantico

git tag -> para listar las etiquetas

git tag numero-version -> crea una etiqueta

git tag -d numero-version -> para eliminar una etiqueta

git show numero-version -> mostrar informacion de una etiqueta

sincronizando la etiqueta del repositorio local al remoto git add . git tag v1.0.0 git commit -m "v1.0.0" git push origin numero-version

generando una etiqueta anotada (con mensaje de commit) git add . git tag -a "v1.0.0" -m "mensaje de la etiqueta" git push --tags

Github Pages

gh-pages es una rama especial para crear un sitio web a tu proyecto alojado directamente en el repositorio de github

url del repositorio: https://github.com/usuario/repositorio

url del sitio: https://usuario.github.io/repositorio

para crear esta rama especial en github ejecutamos los siguientes comandos git branch gh-pages git checkout gh-pages git remote add origin https://url git push origin gh-pages

git pull origin gh-pages -> para descargar cambios del repositorio remoto al local

Colaboracion en github

Para poder colaborar en proyectos alojados en github se necesita hacer uso de los forks y pulls requests.

primero se debe hacer fork al repositorio en el que se quiere colaborar que no es mas que tener una copia del repositorio en el que se quiere colaborar github brinda la opcion en un boton y consulta si quieres dejar el mismo nombre al repositorio y si solo quieres copiar la rama principal

segundo se debe clonar el repositorio para poder manejarlo localmente (tener el repositorio en nuestra computadora) se usa el comando -> git clone + url del repo (posicionados ya en la carpeta donde queremos los archivos)
